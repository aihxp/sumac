# Milestone v1.1: Platform Hardening and Core Expansion

**Status:** Completed 2026-04-08
**Phases:** 7-12
**Total Plans:** 12

## Overview

This milestone extended the internal rewrite beyond the golden path by
hardening `watch`, tightening skill install and serve policy, and moving the
next orchestration-heavy command families behind clearer seams without changing
the shipped `1.x` surface. The delivery order followed the dependency chain
from one canonical managed asset model, through watch runtime hardening and
secure skill staging, into enforced scan and serve policy, then contract-gated
command extraction, and finally a soak-based rollback decision.

## Phases

### Phase 7: Canonical Asset Inventory & Watch Parity

**Goal**: Users can rely on one canonical managed skill asset inventory for
watch invalidation across native-event and polling modes.
**Depends on**: Phase 6
**Plans**: 2 plans

- [x] 07-01: Make the managed skill asset inventory canonical and recursive
- [x] 07-02: Prove nested watch parity in native and polling modes

### Phase 8: Watch Runtime Hardening

**Goal**: Long-running `sxmc watch` sessions stay responsive while notify side
effects and watch orchestration run through a dedicated runtime seam.
**Depends on**: Phase 7
**Plans**: 2 plans

- [x] 08-01: Move `sxmc watch` behind a dedicated runtime seam

### Phase 9: Secure Skill Materialization & Atomic Activation

**Goal**: Users can install or update local and git-backed skills through

**Depends on**: Phase 8
**Plans**: 2 plans

### Phase 10: Unified Scan & Serve Enforcement

**Goal**: Installed skill contents are scanned and served under one enforced

**Depends on**: Phase 9
**Plans**: 2 plans

- [x] 10-01: Align scanning with the canonical managed asset inventory
- [x] 10-02: Serve only approved managed assets from installed skills

### Phase 11: Command-Family Extraction & Contract Gates

**Goal**: `watch` and `skills` run through clearer service boundaries without

**Depends on**: Phase 10
**Plans**: 2 plans

- [x] 11-01: Move the `skills` family behind a dedicated app service

### Phase 12: Soak Evidence & Rollback Decision

**Goal**: The migrated watch and skills route is kept or rolled back based on

**Depends on**: Phase 11
**Plans**: 2 plans

- [x] 12-01: Record the concrete soak-evidence report for the migrated route
- [x] 12-02: Make the rollback decision explicit and archive-ready

## Milestone Summary

### Key Accomplishments

- made managed skill assets the single policy source for watch, install, scan,
  and serve
- moved `sxmc watch`, `skills`, and CLI-facing `serve` onto clearer app seams
- hardened local and git-backed skill activation with staging and allowlists
- aligned serve behavior and scanner coverage to the same managed asset set
- documented the explicit rollback decision and later release-soak requirement

### Issues Resolved

- removed more orchestration-heavy logic from top-level dispatch hotspots
- closed silent-skip behavior in scanner coverage for managed assets
- replaced ad hoc skill activation with staged, atomic updates

### Deferred

- operator-facing diagnostics for watch backend health and degraded mode
- provenance previews and trust policy for remote skill installs
- final retirement of `SXMC_GOLDEN_PATH_ROUTE=legacy` after later release soak
